In a nutshell, htmz lets you load HTML resources into any part of the page with nothing but vanilla HTML code. Imagine clicking a link, but instead of reloading the whole page, it only updates the relevant portion of the page.

Installing

Simply copy the following snippet into your page:

<iframe hidden name=htmz onload="setTimeout(()=>document.querySelector(this.contentWindow.location.hash||':not(*)')?.replaceWith(...this.contentDocument.body.childNodes))"></iframe>

Alternatively, if you do insist on using the npm ecosystem, the following npm commands automate the process of copying the snippet into your page: 

npm install --save-dev htmz
npx htmzify ./path/to/my/index.html

For the development version (deminified), see GitHub repo: github.com/Kalabasa/htmz/blob/master/htmz.dev.html

Basic usage

To invoke htmz, you need a hyperlink (or form) having these attributes:

  • The href (or action) attribute pointing to the resource URL href="/flower.html‚ãØ
  • Also within the href: the destination ID selector immediately after the URL ‚ãØ#my-container"
  • And a target attribute with this value target=htmz 
<!-- Loads /flower.html onto #my-container -->
<a href="/flower.html#my-container" target=htmz>Flower</a>

While this looks like an abuse of the URL fragment (it is), there is no other use for the URL fragment when the response is already an HTML fragment, so it was repurposed as the destination ID selector. And it already looks like a CSS ID selector.

‚ö† Important note: The loaded content replaces the selected destination. It may not be intuitive at first, but htmz does not place the content inside the destination. The rationale is that replacement is a more powerful operation. With replacement, you can replace, delete (replace with nothing), and place-inside (replace with the same container as wrapper).

What does it do exactly?

htmz does one thing and one thing only.

Enable you to load HTML resources within any element in the page.

Think tabbed UIs, dual-pane list-detail layouts, dialogs, in-place editors, and the like. With htmz, you can do these things without having to write client-side scripts or mess with UI frameworks on top of HTML/JS.

This idea is not new. Dividing web pages into independently reloading parts has been a thing since mid-1990s. They were called frames, namely, <iframe>s, <frame>s, and <frameset>s.

htmz is a generalisation of HTML frames. — Load HTML resources within any frame any element in the page.

Advanced usage

Naturally, only <a> and <form> elements can target and invoke htmz (as of current HTML5). This is fine; it‚Äôs semantic, after all. However, HTML offers a couple more features that work well with htmz.

Per-button action & target

If you want to override the form‚Äôs action on a per-button basis, use the <button>‚Äôs formaction attribute. 

<form action="/default#my-target" target=htmz>
  <button>
    Default form action
  </button>
  <button formaction="/button#my-target">
    Different button action
  </button>
  <button formaction="/another-action#another-target">
    Another action
  </button>
</form>

Base target

Tired of adding target=htmz to every link and form?

Using the base element, set htmz as the default target for all relative links. Add this at the top of your page. 

<base target=htmz>

Scripting / interactivity

If you need something more interactive than the request-response model, you may try the htmz companion scripting language: javazcript. Sorry, I meant JavaScript, a scripting language designed to make HTML interactive.

Core htmz will not implement advanced interactions, lest it fall into the trap of feature creep (such as adding a completely new syntax for every new use-case).

htmz does not preclude you writing JS or using UI libraries to enhance user interaction. You could, say, enhance a single form control with vanillaJS, but the form values could still be submitted as a regular HTTP form with htmz. It largely depends on the kind of interactivity required.

That said, htmz is extensible!

FAQ

How does it work?

htmz is an iframe named "htmz". You invoke htmz by loading a URL into the iframe via target=htmz. It has an onload handler to post-process the HTML resources you loaded.

htmz is essentially a proxy target.

Like how a proxy server forwards requests to some specified server, proxy target htmz forwards responses into some specified target. 

<!--
  the main idea
    /flower.html => #my-container
-->
<a href="/flower.html" target="#my-container">Flower</a>

<!--
  actual code
    /flower.html =htmz> #my-container
-->
<a href="/flower.html#my-container" target=htmz>Flower</a>

When you load a URL into the htmz iframe, the onload handler kicks in. It extracts your destination ID selector from the URL hash fragment and transplants the iframe‚Äôs contents (now containing the loaded HTML resource) into your specified destination.

htmz only runs when you invoke it. It does not continually parse your DOM and scan it for special attributes or syntax, nor does it attach listeners in your DOM. It‚Äôs a proxy not a VPN.

What are the limitations?

The main direct limitation is having only one destination per response. However, this can be fixed by writing an extension. ;)

A more general but classic limitation is the request-response model. The Web 1.0 model. This might mean putting more UI logic in the web server. This can be a good thing or a bad thing, as it can mean either consolidation or fragmentation of UI logic, which respectively decreases or increases complexity. It really depends on your goal and style.
