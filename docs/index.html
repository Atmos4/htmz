How do you use it?

To invoke htmz, you need a hyperlink (or form) having these attributes:

  • The href (or action) attribute pointing to the resource URL href="/flower.html‚ãØ
  • Also within the href: the destination ID selector immediately after the URL ‚ãØ#my-container"
  • And a target attribute with this value target=htmz 
<!-- Loads /flower.html into #my-container -->
<a href="/flower.html#my-container" target=htmz>Flower</a>

This looks like an abuse of the URL fragment. And yes, it is. Is this still vanilla HTML? Yep, it still is. Since there is no other use for the URL fragment in this case, and it looks like a CSS ID selector, it was repurposed as the destination ID selector.

What is target=htmz? This attribute tells the browser to load the resource into htmz instead of loading it into the entire window. A section below details how htmz works#.

‚ö† Important note: The loaded content replaces the selected destination. It may not be intuitive at first, but htmz does not place the content inside the destination. The rationale is that replacement is a more powerful operation. With replacement, you can replace, delete (replace with nothing), and place-inside (replace with the same container as wrapper).

Naturally, only <a> and <form> elements can target and invoke htmz (as of current HTML5). This is fine; it‚Äôs semantic. Don‚Äôt forget the formaction attribute, though!

If you need something more dynamic than the request-response model, you may use JavaScript. Try vanilla JS, a great match.

What does it do exactly?

htmz does one thing and one thing only.

Enable you to load HTML resources within any element in the page.

Imagine clicking a link, but instead of reloading the whole page, it only updates a relevant portion of the page. Think tabbed UIs, dual-pane list-detail layouts, dialogs, in-place editors, and the like. Imagine doing these things without having to write client-side scripts or mess with UI frameworks on top of HTML/JS.

This idea is not new. Dividing web pages into independently reloading parts has been a thing since mid-1990s. They were called frames, namely, <iframe>s, <frame>s, and <frameset>s.

htmz is a generalisation of HTML frames. — Load HTML resources within any frame any element in the page.

Base target

Tired of adding target=htmz to every link and form?

Using the base element, set htmz as the default target for all relative links. Add this at the top of your page. 

<base target=htmz>

FAQ

How does it work?

htmz is an iframe named "htmz". You invoke htmz by loading a URL into the iframe via target=htmz. It has an onload handler to post-process the HTML resources you loaded.

htmz is essentially a proxy target.

Like how a proxy server forwards requests to some specified server, proxy target htmz forwards responses into some specified target. 

<!--
  the main idea
    /flower.html => #my-container
-->
<a href="/flower.html" target="#my-container">Flower</a>

<!--
  actual code
    /flower.html =htmz> #my-container
-->
<a href="/flower.html#my-container" target=htmz>Flower</a>

When you load a URL into the htmz iframe, the onload handler kicks in. It extracts your destination ID selector from the URL hash fragment and transplants the iframe‚Äôs contents (now containing the loaded HTML resource) into your specified destination.

Unlike other frameworks, htmz only runs when you invoke it. It does not continually scan your DOM for special attributes or syntax, nor does it attach listeners anywhere. It‚Äôs a proxy not a VPN.

What are the limitations?

The main direct limitation is having only one destination per response. However, this can be fixed by writing an extension. ;)

A more general but classic limitation is the request-response model. The Web 1.0 model. This might mean putting more UI logic in the web server. This can be a good thing or a bad thing, depending on your goal.

But just like HTML, htmz does not preclude you writing JS or using UI libraries to enhance user interaction. You could, say, enhance a single form control with client-side scripting, but the form values would still be submitted as a regular form with htmz. It largely depends on the kind of app and interactivity required.
